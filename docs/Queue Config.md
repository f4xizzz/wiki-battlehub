# **Queue Config**

---

## **Matchmaking Queue Configuration**

The `queue_config.json` file controls the matchmaking rules for **Cobblemon BattleHUB**, focusing primarily on preventing result manipulation (Win Trading) through configurable cooldowns between the same players.

---

### **File Path**

`config/cobblemon\_battlehub/queue\_config.json`

## **Default Configuration Template**

Below is the default JSON template automatically generated by the mod:

        {  
         "QueueAntiTradingCasual": 30,  
         "QueueAntiTradingRanked": 60  
        }

---

## **Detailed Parameter Explanation**

### **1. Win Trading Prevention (Casual Queue)**

* **QueueAntiTradingCasual** (Default: 30):  
  The wait time (in minutes) that two specific players must wait before they can be matched again in the **Casual** queue after fighting each other. This prevents friends from abusing fast matchmaking to intentionally play consecutive matches in the public queue.

### **2. Win Trading Prevention (Ranked Queue)**

* **QueueAntiTradingRanked** (Default: 60):  
  The wait time (in minutes) that two specific players must wait before they can be matched again in the **Ranked** queue after a competitive duel. Since ranked matches award rating points and rewards, this limit is higher (1 hour by default) to prevent artificial point gains (Elo boosting).

---

## **How the Matchmaking System Works (Behind the Scenes)**

The Cobblemon BattleHUB algorithm performs several automatic safety checks when a player attempts to enter the queue:

### **Penalty Check (Leaver Buster)**

If a player tries to enter the Ranked queue but has a recent history of intentional disconnects (quits), the system blocks entry immediately:

1. The mod checks whether the timeout for the desertion ban is still active.  
2. If it is active, the player receives a chat alert indicating the exact remaining suspension time formatted in hours and minutes.

### **Rating Calculation (Glicko-2)**

For competitive matches, the system does not perform random searches. It dynamically calculates:

* **Rating:** The player’s current skill score (default starting value 1500.0).  
* **RD (Rating Deviation):** The reliability of the player’s skill estimate (default starting value 350.0). Matchmaking progressively expands acceptable Elo difference limits as queue wait time increases.

### **Available Arena Validation**

The queue will only match players if there is at least **one empty physical arena** available in the battle dimension. If all arenas are occupied, players remain in queue waiting until a match finishes.

---

## **How to Reload Changes**

If you modify the anti-trading minute values in `queue_config.json`, apply the changes to the server in real time using the command:

`/bh reload`
